# スパン F1（entity-level スパン F1）

[English version](span-f1.md)

---

## これは何？

**スパン F1（entity-level スパン F1）** は、住所を部品に切り出す精度を **「部品（スパン）単位」**で測る評価指標です。
予測したスパンは、**ラベルが一致し、かつ開始位置と終了位置が正解とぴったり一致したときだけ**正解とみなします。
そのうえで適合率（precision）・再現率（recall）・F1 をスパン全体について計算します。

たとえるなら、答案を **「文字ごとの○×」ではなく「単語まるごと合っているか」** で採点する先生です。
「丸の内」を「丸の」「内」と切ってしまったら、文字単位ではほとんど合っていても、
**スパンとしては不正解**——境界を 1 つ間違えただけで部品ごと×になります。これが本当の実力を映します。

---

## なぜ重要なのか？

文字ごとの正解率（token accuracy）は、住所の評価では **甘すぎて当てになりません**。
`O`（どの部品でもない）が多い住所では、全部 `O` と答えるだけでも正解率が高く出てしまいます。

実際 kugiri の `SynthDemo` は token accuracy **1.000** を出しますが、これは
**合成データが規則的すぎる**ためで、**実力ではありません**（[合成データ](synthetic-data.ja.md) 参照）。
スパン F1 は「部品をまるごと正しく切れたか」を問うので、こうした見かけ倒しを暴きます。
だから kugiri の完了の定義は「**スパン F1 で評価し、合成 1.000 に依存しない数値で示す**」ことです。

---

## どう動くのか？

1. 予測の [BIOES](bioes.ja.md) ラベル列を **スパンの集合**にデコードする（`B-丁目 I-丁目 E-丁目` → `丁目:[3,6)`）。
2. 正解も同じくスパン集合にする。
3. **ラベル＋開始＋終了が完全一致**したスパンだけを TP（真陽性）として数える。

```
  正解スパン:  [都道府県:0-3] [市:3-7]   [町:7-10]      [丁目:10-13]
  予測スパン:  [都道府県:0-3] [市:3-7]   [町:7-9] [?:9-10]  [丁目:10-13]
                  ✓TP          ✓TP        ✗(境界ズレ→FP+FN)   ✓TP

  precision = TP / (TP + FP)        ← 予測のうち正しかった割合
  recall    = TP / (TP + FN)        ← 正解のうち拾えた割合
  F1        = 2PR / (P + R)         ← 両者の調和平均
```

`町:7-10` を `7-9` と切った 1 文字のズレだけで、その部品は丸ごと不正解（FP かつ FN）になります。

---

## kugiri ではどう使われている？

実装は `eval/SpanEval.java`（entity-level スパン P/R/F1・混同行列・hold-out 評価）と、
公開窓口 `tagger/AddressParser.evaluateSpans`。`demo/EvalDemo` が実演します。

評価は必ず **hold-out**（学習に使っていないデータ）で取ります。合成データの 1.000 ではなく、
**per-label のスパン F1** を出すことが、変更を測る正しい物差しです（[字推定](aza.ja.md) の
内部境界 F1=0.829 もこの考え方で採点しています）。
