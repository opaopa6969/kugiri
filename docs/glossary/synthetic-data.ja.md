# 合成データ（Synthetic Data）

[English version](synthetic-data.md)

---

## これは何？

合成データとは、**実データを集める代わりに、正解の作り方が分かっている部品から
人工的に学習用サンプルを組み立てる**手法です。

kugiri では、住所をいきなり文字列として持つのではなく、
**コンポーネント列 `[(ラベル, 表層), …]`** として持ちます。
これを連結すれば文字列ができ、**同時に** codepoint 単位の [BIOES](bioes.ja.md) ラベルが
**アライメント計算なしで**確定します。

```
[(都道府県,東京都),(東京23区,千代田区),(町,丸の内),(丁目,一丁目),(街区符号,9番),(住居番号,1号)]
        │ 連結
        ▼
  "東京都千代田区丸の内一丁目9番1号"
        │ 連結位置から
        ▼
  B-都道府県 I-都道府県 E-都道府県 B-東京23区 ... S-住居番号  ← 正解が自動で付く
```

レゴブロックを組むようなもので、**どのブロックが何色かは最初から分かっている**ので、
組み上がった作品のラベルも自動で決まります。

---

## なぜ重要なのか？

系列ラベラ（[パーセプトロン](perceptron-tagger.ja.md)など）を学習するには、
**文字1つ1つにラベルが付いた正解**が大量に要ります。実データはこのラベルが付いていません。

合成データなら、ラベル付き正解を**無限に・正確に**作れるので、
配線（コードが端から端まで動くか）の確認や、素性・モデルの初期検証にうってつけです。
[弱教師](weak-supervision.ja.md) が頭しかラベルできないのに対し、合成データは**完全教師**です。

---

## どう動くのか？

`synth/Synth.augment()` が、ラベルを保ったまま**表層だけ**を揺らしてデータ拡張します。
実入力の表記揺れに強くするためです。

```
  同じ意味・同じラベルのまま、見た目だけ変える：

    数字     全角 ⇄ 半角        一 ⇄ 1
    ハイフン  - ー − － ‐        異字をランダムに
    丁目/番地/号  → 記号化       1丁目 ⇄ 1-
    ZIP      〒 の有無           〒020-0021 ⇄ 020-0021

  ★ ラベルは保持されるので、拡張後も正解アライメントが壊れない
```

### ⚠️ 「精度 1.000」を信じてはいけない

`SynthDemo` は tag accuracy **1.000** を出しますが、これは**合成データが規則的すぎる産物**であり、
**実力ではありません**。実評価は必ず実ラベルの hold-out で
[スパン F1](span-f1.ja.md)（entity-level）を取ること。これは DESIGN/HANDOFF でも繰り返し警告されています。

---

## kugiri ではどう使われている？

合成器は `synth/Synth.java`（`buildExample()` / `fromRecords()` / `augment()`）に実装され、
`demo/SynthDemo.java` が end-to-end（合成→学習→parse）を実演します。

`fromRecords()` は [弱教師](weak-supervision.ja.md) で作った実データ由来の Component 列からも
学習サンプルを生成できるため、合成と実データの両方を同じ経路で扱えます。
