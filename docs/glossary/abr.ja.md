# ABR（アドレス・ベース・レジストリ）

[English version](abr.md)

---

## これは何？

**ABR（アドレス・ベース・レジストリ / Address Base Registry）** は、日本の住所を整備した
公的な「住所マスタ」です。デジタル庁が整備し、町字・街区・住居番号・地番が
それぞれ ID で分解された形で配られます。

図書館の蔵書目録にたとえると分かりやすいです。本を「背表紙の文字列」として持つのではなく、
著者・タイトル・出版社…と **項目ごとに分解されたカード** として持つのが ABR です。
住所を「丸ごとの文字列」ではなく、`大字 / 丁目 / 字 / 街区符号 / 住居番号` の
**項目に割れた状態** で受け取れる、というのが kugiri にとっての価値です。

---

## なぜ重要なのか？

kugiri は住所を [階層要素](hierarchy.ja.md) ごとの token 列に切り出す学習器です。
学習には「どこからどこまでが何の階層か」という正解ラベルが要りますが、
未分割の生住所文字列にはそれがありません。

ABR は **すでに項目ごとに割れている** ので、住所の **頭（大字〜住居番号, lv4–7）** について
ラベル付き教師データを、人手アノテーションなしで自動生成できます。これが
[弱教師](weak-supervision.ja.md) の土台になります。そして頭が固まることで、
末尾に残る正解の無いスパン＝[字](aza.ja.md) を教師なしで推定する足場が整います。

---

## どう動くのか？

ABR は ZIP（郵便番号）を持たないので、kugiri は ZIP を持つ
[KEN_ALL](ken-all.ja.md) で **大まかに引いてから、その町域を ABR で引き直す**
（通称「opa 方式」）。突合は団体コードではなく **文字列** で行います。
KEN_ALL の団体コードは 5 桁、ABR は 6 桁で、そのまま結合すると合わないからです。

```
  ① ZIP で KEN_ALL を引く
     020-0021 → 岩手県 / 盛岡市 / 上田
                          │ 町域名で突合（文字列）
                          ▼
  ② ABR で引き直す（項目に割れている）
     大字=上田  丁目=…  街区符号=…  住居番号=…

  ★ KEN_ALL の連結フィールドが ABR で割れる：
     千代田区        → 東京23区:千代田区
     横浜市西区      → 市:横浜市   + 区:西区
     虻田郡洞爺湖町  → 群:虻田郡   + 町村:洞爺湖町
```

ABR の列は **ヘッダ名で参照** して読みます（`get("列名")`）。実ダンプの版で列が増減しても、
コードを直すのは **列名の文字列だけ** で済みます。

---

## kugiri ではどう使われている？

実装は `abr/Abr.java`（KEN_ALL→ABR 再解決して [階層要素](hierarchy.ja.md) の Component 列を作る）と
`abr/Csv.java`（ヘッダ名参照の CSV 読み）です。`demo/AbrDemo` が
東京23区 / 市+区 / 群+町村 の分離を実演します。

> なお、実 ABR/KEN_ALL の生データはリポジトリにコミットしません（サンプルのみ）。
> 実データ取込は実スキーマに合わせて列名を検証してから行います。
